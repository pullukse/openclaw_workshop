# HEARTBEAT.md Example

## Purpose

This file gives the agent instructions for proactive wakeups.

When a heartbeat runs, the agent can use this file to decide what useful background work to do.

A good `HEARTBEAT.md` is short, practical, and tied to real priorities.
It should not become a giant wish list.

---

# HEARTBEAT.md

## Priority checks

Rotate through these during the day when useful:
- check for urgent unread email
- check calendar events in the next 24 hours
- check whether any active project has a clear next step

## Proactive work

When there is no urgent interruption, prefer useful background progress such as:
- cleaning up docs
- updating memory files
- organizing current project folders
- tightening draft materials that are already in progress

## When to notify the user

Only interrupt when there is something worth surfacing:
- an important email
- an upcoming event that matters soon
- a completed deliverable
- a real blocker
- a decision that needs user input

## When to stay quiet

Stay quiet when:
- there is nothing materially new
- it is late and nothing is urgent
- the last check was recent and produced no useful change
- you are better off doing silent background work

## Special reminder

Do not use heartbeat just to report status.
Use it to create progress.

---

## How to customize this file

Students should customize heartbeat instructions based on:
- what periodic checks matter
- how proactive they want the agent to be
- what types of updates are worth interrupting for
- what background work is actually useful

### Good things to include
- a few recurring checks
- one or two useful proactive tasks
- clear notification rules

### Things to avoid
- huge task lists
- vague goals with no action
- repetitive status-report instructions

### Good rule of thumb
If a heartbeat prompt does not help the agent make a useful decision, it is too vague.
